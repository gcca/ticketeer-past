# Ticketeer API

Documentation for the endpoints implemented in `src/ticketeer/handling/`.

## Base URL

The relative paths of the endpoints start from /ticketeer/api/

By default, the server listens on:

```text
http://localhost:5521
```

The bind and port can be changed with the binary options:

```sh
./ticketeer-api --bind 0.0.0.0 --port 5521
```

## General Format

The API uses JSON for requests and responses when there is a body.

Recommended headers:

```http
Content-Type: application/json
Accept: application/json
```

Authenticated endpoints expect a Bearer token:

```http
Authorization: Bearer ticketeer-v1_<token>
```

IDs are returned as strings in dashboard endpoints, even though they are `bigint` in the database.

## Enums

Possible values for attributes backed by enums in the database:

### `helpdesk.role`

Appears as `profile.role` in `GET /ticketeer/api/dashboard/v1/landing`.

| Value | Description |
| --- | --- |
| `administrator` | Helpdesk administrator |
| `supervisor` | Ticket/team supervisor |
| `technician` | Technician assignable to tickets |
| `requester` | Requesting user |

### `helpdesk.ticket_trait`

Appears as `ticket_statuses[].trait` in `GET /ticketeer/api/dashboard/v1/landing`.

| Value | Description |
| --- | --- |
| `open` | Open state |
| `in_progress` | In progress state |
| `closed` | Closed state |

### `helpdesk.ticket_activity_kind`

Appears as `activities[].kind` in `GET /role/requester/v1/ticket/{ticket_id}/details` and `GET /role/requester/v1/ticket/{ticket_id}/activity/list`.

| Value | Description |
| --- | --- |
| `message` | Activity message |
| `status_changed` | Ticket status change |
| `assigned_changed` | Assignment change |
| `department_changed` | Department change |
| `priority_changed` | Priority change |
| `due_date_changed` | Due date change |

### `auth.provider`

Not directly exposed in current responses. Used internally in `POST /ticketeer/api/auth/v1/signin/overlord` to save the user's OAuth relationship.

| Value | Description |
| --- | --- |
| `overlord` | User authenticated from Overlord |

## Errors

Errors return a JSON object with this format:

```json
{
  "message": "Error message"
}
```

Common errors:

| Code | Case |
| --- | --- |
| `400` | Invalid JSON, missing parameters, or required header/cookie absent |
| `401` | Invalid credentials or invalid/expired token |
| `500` | Internal failure when creating session or loading profile |
| `503` | Main database unavailable |

## POST `/ticketeer/api/auth/v1/signin`

Logs in with username and password stored locally in `auth.user`.

Full URL:

```text
/ticketeer/api/auth/v1/signin
```

### Request

```http
POST /ticketeer/api/auth/v1/signin
Content-Type: application/json
```

```json
{
  "username": "administrator",
  "password": "t801"
}
```

Contract:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `username` | string | Yes | Local username |
| `password` | string | Yes | Plain text password sent by the client |

### Response `200`

The created session expires one month after its creation.

Header:

```http
Authorization: Bearer ticketeer-v1_012345...
```

Body:

```json
{
  "token": "ticketeer-v1_0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "expiry": "2026-05-26 05:30:00.000000+00"
}
```

Contract:

| Field | Type | Description |
| --- | --- | --- |
| `token` | string | Session token generated by Ticketeer |
| `expiry` | string | Expiration timestamp returned by PostgreSQL |

### Errors

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Invalid JSON body"}` | The body is not valid JSON |
| `400` | `{"message":"Missing username or password"}` | Missing `username` or `password`, or they are not strings |
| `401` | `{"message":"Invalid credentials"}` | Username/password do not match |
| `500` | `{"message":"Session creation failed"}` | Could not insert the session |
| `503` | `{"message":"Database unavailable"}` | Could not connect to `DB_URL` |

### Example

```sh
http POST http://localhost:5521/ticketeer/api/auth/v1/signin username=administrator password=t801
```

## POST `/ticketeer/api/auth/v1/signin/overlord`

Logs in using an existing Django session from the Overlord system. The endpoint reads the `sessionid` cookie, validates the session in `OVERLORD_DB_URL`, resolves or creates a local user with the `_overlord` suffix, and returns a Ticketeer-specific token.

Full URL:

```text
/ticketeer/api/auth/v1/signin/overlord
```

### Request

```http
POST /ticketeer/api/auth/v1/signin/overlord
Cookie: sessionid=<django-session-key>
```

Does not require body.

Contract:

| Field | Location | Type | Required | Description |
| --- | --- | --- | --- | --- |
| `sessionid` | Cookie | string | Yes | Django session key in the Overlord database |

### Response `200`

Header:

```http
Authorization: Bearer ticketeer-v1_012345...
```

Body:

```json
{
  "token": "ticketeer-v1_0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "expiry": "2026-05-26 05:30:00.000000+00"
}
```

Contract:

| Field | Type | Description |
| --- | --- | --- |
| `token` | string | Session token generated by Ticketeer |
| `expiry` | string | Expiration timestamp returned by PostgreSQL |

### Errors

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Missing session"}` | Missing `sessionid` cookie, Django session does not exist, expired, does not contain `_auth_user_id`, or could not resolve the local user |
| `500` | `{"message":"Session creation failed"}` | Could not insert the Ticketeer session |
| `503` | `{"message":"Database unavailable"}` | Could not connect to `DB_URL` |

### Example

```sh
http POST http://localhost:5521/ticketeer/api/auth/v1/signin/overlord Cookie:sessionid=django_session_key
```

## Basic Usage Flow

```sh
# 1. Log in with local credentials.
TOKEN="$(http -b POST http://localhost:5521/ticketeer/api/auth/v1/signin username=administrator password=t801 | jq -r '.token')"

# 2. Validate the current session.
http GET http://localhost:5521/ticketeer/api/auth/v1/me Authorization:"Bearer $TOKEN"

# 3. Load initial dashboard data.
http GET http://localhost:5521/ticketeer/api/dashboard/v1/landing Authorization:"Bearer $TOKEN"

# 4. Log out.
http POST http://localhost:5521/ticketeer/api/auth/v1/signout Authorization:"Bearer $TOKEN"
```

## GET `/role/requester/v1/ticket/list`

Returns the list of tickets for the authenticated user (based on their profile).

Full URL:

```text
/ticketeer/api/role/requester/v1/ticket/list
```

### Request

```http
GET /ticketeer/api/role/requester/v1/ticket/list?s=laptop
Authorization: Bearer ticketeer-v1_<token>
```

Contract:

| Field | Location | Type | Required | Description |
| --- | --- | --- | --- | --- |
| `Authorization` | Header | string | Yes | Must have format `Bearer <token>`. The user must have role `requester`. |
| `s` | Query | string | No | Search text. Currently filters tickets whose description or activity body contains this value. |

### Response `200`

```json
[
  {
    "id": "1",
    "description": "The laptop does not turn on after a BIOS update.",
    "status_id": "3",
    "created_at": "2024-04-26T06:14:39Z",
    "activities": [
      {
        "body": "Received. We will remove the equipment to check power supply, battery and motherboard.",
        "created_at": "2024-04-26T08:14:39Z"
      }
    ]
  }
]
```

Note: the example shows an abbreviated array. The real response includes at
most 57 user tickets ordered by creation date descending. `activities` is
ordered by `created_at` descending.

Contract for each ticket:

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | Ticket ID |
| `description` | string | Ticket description |
| `status_id` | string | Ticket status ID |
| `created_at` | string | Creation timestamp |
| `activities` | array | Preview of ticket activities |
| `activities[].body` | string | Activity text |
| `activities[].created_at` | string | Activity timestamp |

### Errors

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Missing Authorization"}` | Missing `Authorization` header or does not start with `Bearer ` |
| `403` | `{"message":"Forbidden: insufficient permissions"}` | The token does not exist, expired or the user does not have role `requester` |
| `503` | `{"message":"Database unavailable"}` | Could not connect to `DB_URL` |

### Example

```sh
TOKEN="$(http -b POST http://localhost:5521/ticketeer/api/auth/v1/signin username=requester01 password=t801 | jq -r '.token')"

http GET http://localhost:5521/ticketeer/api/role/requester/v1/ticket/list Authorization:"Bearer $TOKEN"
```

## Supervisor Role Endpoints

The supervisor role currently exposes endpoints mostly equivalent to the
requester role endpoints. They require a valid Ticketeer token and a local
profile with role `supervisor`.

The request and response contracts are the same as the corresponding
`/role/requester/...` endpoints documented above. Replace the path prefix:

```text
/ticketeer/api/role/requester
```

with:

```text
/ticketeer/api/role/supervisor
```

Exception: `GET /ticketeer/api/role/supervisor/v1/ticket/list` returns at most
57 tickets ordered by creation date descending. The requester equivalent only
returns tickets created by the authenticated requester. Both list endpoints
accept the optional query parameter `s`, which currently filters tickets whose
description or activity body contains the given value.

Available endpoints:

| Method | Path | Equivalent requester endpoint |
| --- | --- | --- |
| `GET` | `/ticketeer/api/role/supervisor/v1/ticket/list` | `GET /ticketeer/api/role/requester/v1/ticket/list` |
| `POST` | `/ticketeer/api/role/supervisor/v1/ticket/create` | `POST /ticketeer/api/role/requester/v1/ticket/create` |
| `GET` | `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/details` | `GET /ticketeer/api/role/requester/v1/ticket/{ticket_id}/details` |
| `GET` | `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/activity/list` | `GET /ticketeer/api/role/requester/v1/ticket/{ticket_id}/activity/list` |
| `POST` | `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/activity/create/message` | `POST /ticketeer/api/role/requester/v1/ticket/{ticket_id}/activity/create/message` |
| `POST` | `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/create` | Supervisor only |
| `GET` | `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/{attachment_id}/download` | Supervisor only |

Supervisor `GET /v1/ticket/{ticket_id}/details` and
`GET /v1/ticket/{ticket_id}/activity/list` include an `attachments` array on
each activity:

```json
{
  "id": "3",
  "kind": "message",
  "body": "Received. We will remove the equipment to check power supply, battery and motherboard.",
  "created_at": "2024-04-26T08:14:39Z",
  "profile_name": "Jill Valentine",
  "attachments": [
    {
      "id": "7",
      "file_name": "invoice.pdf"
    }
  ]
}
```

Attachment contract:

| Field | Type | Description |
| --- | --- | --- |
| `activities[].attachments` | array | Files attached to the activity |
| `activities[].attachments[].id` | string | Attachment ID for constructing download URLs |
| `activities[].attachments[].file_name` | string | Uploaded file name |

Download URLs can be constructed using the pattern: `/ticketeer/api/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/{attachment_id}/download`

## POST `/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/create`

Uploads a file and creates a `helpdesk.ticket_activity_attachment` record for
an existing ticket activity. The file is stored under `UPLOAD_DIR` using this
relative path:

```text
role/ticket_id={ticket_id}/activity_id={activity_id}/attachment_id={attachment_id}
```

Full URL:

```text
/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/create
```

### Request

```http
POST /ticketeer/api/role/supervisor/v1/ticket/1/activity/3/attachment/create
Authorization: Bearer ticketeer-v1_<token>
Content-Type: multipart/form-data
```

Path parameters:

| Parameter | Type | Description |
| --- | --- | --- |
| `ticket_id` | string | Ticket ID |
| `activity_id` | string | Ticket activity ID. The activity must belong to the ticket. |

Contract:

| Field | Location | Type | Required | Description |
| --- | --- | --- | --- | --- |
| `Authorization` | Header | string | Yes | Must have format `Bearer <token>`. The user must have role `supervisor`. |
| `file` | Form data | file | Yes | Uploaded file. Maximum size is 5 MB. |

The server infers `mime_type` from the uploaded file extension. Unknown
extensions use `application/octet-stream`.

### Response `200`

```json
{
  "id": "7",
  "ticket_activity_id": "3",
  "file_path": "role/ticket_id=1/activity_id=3/attachment_id=7",
  "file_name": "invoice.pdf",
  "file_size": "104857",
  "mime_type": "application/pdf",
  "created_at": "2024-04-26T08:14:39Z"
}
```

Contract:

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | Created attachment ID |
| `ticket_activity_id` | string | Activity ID that owns the attachment |
| `file_path` | string | Relative storage path without the file name |
| `file_name` | string | Uploaded file name |
| `file_size` | string | Uploaded file size in bytes |
| `mime_type` | string | MIME type inferred from the extension |
| `created_at` | string | Creation timestamp |

### Errors

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Missing Authorization"}` | Missing `Authorization` header or does not start with `Bearer ` |
| `400` | `{"message":"Content-Type must be multipart/form-data"}` | Request is not multipart form data |
| `400` | `{"message":"Invalid multipart form-data"}` | Multipart body could not be parsed |
| `400` | `{"message":"Missing file"}` | No uploaded file was sent |
| `400` | `{"message":"Missing file name"}` | Uploaded file does not have a file name |
| `413` | `{"message":"File exceeds 5MB limit"}` | Uploaded file is larger than 5 MB |
| `403` | `{"message":"Forbidden: insufficient permissions"}` | The token does not exist, expired or the user does not have role `supervisor` |
| `404` | `{"message":"Ticket activity not found"}` | The activity does not exist or does not belong to the ticket |
| `400` | `{"message":"Failed to save attachment"}` | Could not write the file to disk |
| `400` | `{"message":"Failed to create attachment"}` | Could not reserve or insert the attachment record |
| `503` | `{"message":"Database unavailable"}` | Could not connect to `DB_URL` |

## GET `/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/{attachment_id}/download`

Downloads an attachment file. The endpoint requires a supervisor token and
verifies that the attachment belongs to the requested activity and ticket.

Full URL:

```text
/role/supervisor/v1/ticket/{ticket_id}/activity/{activity_id}/attachment/{attachment_id}/download
```

### Request

```http
GET /ticketeer/api/role/supervisor/v1/ticket/1/activity/3/attachment/7/download
Authorization: Bearer ticketeer-v1_<token>
```

Path parameters:

| Parameter | Type | Description |
| --- | --- | --- |
| `ticket_id` | string | Ticket ID |
| `activity_id` | string | Ticket activity ID |
| `attachment_id` | string | Attachment ID |

### Response `200`

Returns the file as an attachment. The content type is the stored `mime_type`.

### Errors

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Missing Authorization"}` | Missing `Authorization` header or does not start with `Bearer ` |
| `403` | `{"message":"Forbidden: insufficient permissions"}` | The token does not exist, expired or the user does not have role `supervisor` |
| `404` | `{"message":"Attachment not found"}` | The attachment does not exist or does not belong to the activity and ticket |
| `404` | `{"message":"Attachment file not found"}` | The database record exists but the file is missing on disk |
| `503` | `{"message":"Database unavailable"}` | Could not connect to `DB_URL` |

### Authorization

```http
Authorization: Bearer ticketeer-v1_<token>
```

The token must be valid and the authenticated user must have role
`supervisor`.

Common authorization errors:

| Code | Body | Cause |
| --- | --- | --- |
| `400` | `{"message":"Missing Authorization"}` | Missing `Authorization` header or does not start with `Bearer ` |
| `401` | `{"message":"Invalid or expired token"}` | The token does not exist or has expired |
| `403` | `{"message":"Forbidden: insufficient permissions"}` | The user does not have role `supervisor` |

### Examples

```sh
TOKEN="$(http -b POST http://localhost:5521/ticketeer/api/auth/v1/signin username=supervisor01 password=t801 | jq -r '.token')"

http GET http://localhost:5521/ticketeer/api/role/supervisor/v1/ticket/list Authorization:"Bearer $TOKEN"

http GET http://localhost:5521/ticketeer/api/role/supervisor/v1/ticket/1/details Authorization:"Bearer $TOKEN"

http POST http://localhost:5521/ticketeer/api/role/supervisor/v1/ticket/1/activity/create/message Authorization:"Bearer $TOKEN" message="Supervisor follow-up"

http --form POST http://localhost:5521/ticketeer/api/role/supervisor/v1/ticket/1/activity/3/attachment/create Authorization:"Bearer $TOKEN" file@./invoice.pdf

http GET http://localhost:5521/ticketeer/api/role/supervisor/v1/ticket/1/activity/3/attachment/7/download Authorization:"Bearer $TOKEN"
```
